# Consumer-Driven Contract Testing

Pact contracts sit between the schema suite and the live suites: schemas assert the shape of a response the platform happened to return, contracts assert that the platform still answers **the requests this framework actually makes**, and prove it against a running provider.

## Who the consumer is, and why

A pact is only meaningful if it is generated by the consumer's own code running against a mock provider. That constraint decides the design.

The obvious framing — "the Report service consumes Booking and Room" — is not available to us. We do not own the Report service and cannot execute its code, so any pact published in its name would be a contract invented on behalf of software we have never run. It would look like evidence and carry none.

What we do own is the client layer in [`src/services/`](../src/services/), and it is a real consumer with real expectations. So:

| Consumer        | Providers                             |
| --------------- | ------------------------------------- |
| `rbp-api-tests` | `rbp-auth`, `rbp-room`, `rbp-booking` |

Every consumer test drives the production `AuthService` / `RoomService` / `BookingService` through the production `HttpClient` against Pact's mock server. No test double, no re-implementation — `HttpClient` already takes its base URL by constructor, so pointing it at the mock required no change to `src/`.

The responses are then validated by the **same Zod schemas** the live suite uses, and the adapters that normalize the two targets (`extractToken`, `createdBooking`) run on the mock's response exactly as they run on the platform's. A pact example that the schema rejects fails the consumer test before it can ever be published.

## What is under contract

Nine interactions, each carrying a provider state:

| Provider      | Interaction                                   | State                          |
| ------------- | --------------------------------------------- | ------------------------------ |
| `rbp-auth`    | login with valid credentials → session cookie | an admin account exists        |
| `rbp-auth`    | login with an incorrect password → rejected   | an admin account exists        |
| `rbp-auth`    | validate a live token                         | an active admin session exists |
| `rbp-room`    | list every room                               | a room exists                  |
| `rbp-room`    | fetch one room by id                          | a room exists                  |
| `rbp-room`    | create a room as an authenticated caller      | an active admin session exists |
| `rbp-booking` | list the bookings held against a room         | a room with one booking exists |
| `rbp-booking` | fetch one booking by id                       | a room with one booking exists |
| `rbp-booking` | create a booking against an existing room     | a room exists                  |

## Provider states are seeded through the real API

RBP has no state-setup endpoint, so the states are established the only way available: by calling the platform. Each handler in [`scripts/verify-providers.ts`](../scripts/verify-providers.ts) provisions what the interaction needs — a session, a room, a booking — reusing the same `provisionRoom` and `adminToken` helpers the live suites use, and returns the identifiers it created.

Those identifiers are injected back into the replayed request through Pact's provider-state generators, so nothing is pinned to a seeded id:

```ts
.withRequest({ method: 'GET', path: fromProviderState('/${roomid}', '/1') })
```

`/1` is only the example the mock server serves during the consumer test. At verification time the value comes from the room the handler just created.

The same mechanism carries the session: `fromProviderState('token=${token}', …)` on the `Cookie` header. This is worth stating because its failure mode is quiet — an expression whose value a handler does not return is replayed **literally**, so the provider receives `token=${token}` and answers `403`. That is precisely how the first `rbp-booking` run failed here: the `a room with one booking exists` handler returned the room and booking ids but no token.

## Which target the contracts describe

Live and the containers are [different versions of the same API](target-differences.md), so a single pact cannot describe both. The contracts describe the **dockerized** provider, and every status in them comes from the same profile the rest of the suite uses:

```ts
expectedStatus('auth.rejected', CONTRACT_TARGET) // CONTRACT_TARGET = 'local'
```

Nothing is hard-coded twice. Change a status in `src/profiles/target-profile.ts` and the pact regenerates; if the provider disagrees, verification fails.

### The drift is measurable, not just documented

Verifying the `rbp-auth` pact against the hosted platform fails, and it fails on exactly the differences recorded in [target-differences.md](target-differences.md):

```
1) a login with an incorrect password
     has status code 403 — expected 403 but was 401
2) a login with valid credentials
     includes header 'set-cookie' — expected 'token=…; Path=/' but was ''
```

The first is the `auth.rejected` row of the status table. The second is the `auth.tokenInBody` capability: live returns the token in a JSON body, the containers set a cookie. The prose table and the contract agree, and the contract is the one a machine can check.

That run also reported a third failure — `validate` answering `403` — which is **not** evidence of drift: the throwaway state handler used for the experiment returned a fabricated token, so live was right to reject it.

## The flow

```
npm run test:pact          consumer tests → pacts/  (hermetic, no platform needed)
npm run pact:broker:up     ephemeral broker + postgres
npm run docker:up          the six RBP services
npm run pact:publish       pacts → broker, versioned by git sha and branch
npm run pact:verify        replay every pact against the running providers,
                           publishing the results back to the broker
npm run pact:can-i-deploy  refuse to proceed unless every result is present and green
```

`can-i-deploy` is a real gate, not a report: asked about a version with no published verification it prints `Computer says no` and exits `1`.

The gate was exercised deliberately rather than assumed. Tightening one response matcher — `roomName` from any string to `/^R-\d+$/` — produced exactly the shape of regression the layer exists to catch: the mock satisfied it, so all nine consumer tests stayed green (a consumer cannot see its provider, which is the whole point), while `pact:verify` exited `1` naming `$.rooms[0].roomName -> Expected '101' to match '^R-\d+$'` and `can-i-deploy` then refused. A gate that has never been observed failing is a decoration.

Consumer tests need nothing running, so `npm run test:pact` is the fast inner loop. Only verification needs containers.

## The broker

[`docker-compose.pact.yml`](../docker-compose.pact.yml) runs `pactfoundation/pact-broker` against Postgres, published on `localhost:9292`. It is ephemeral and unauthenticated by design: it exists for the length of a CI job or a local session, is bound to the loopback interface, and holds nothing but contracts already in this repository. `npm run pact:broker:down` drops the volume with it.

Two containers talk to it and they do not share an address. The verifier runs on the host and reaches it at `http://localhost:9292`; the `pact-cli` container reaches it at `http://pact-broker:9292` over the compose network. Using the service name rather than `--network host` keeps the commands identical on macOS and on a Linux runner — host networking is not equivalent across the two, and that difference has already cost this project one silently broken workflow ([security-scan.md](security-scan.md)).

## In CI

The `Contract tests (Pact)` job in [`ci.yml`](../.github/workflows/ci.yml) runs the whole flow on every pull request. It gates merges, and unlike the live suites it depends on nothing outside the runner: containers only, no shared demo environment, no network to a third party. That makes it the most deterministic signal in the pipeline.

The consumer version is the commit sha and the branch is the PR branch, so the broker records which commit last satisfied which provider.

## Adding an interaction

1. Add it to the relevant `tests/pact/*.consumer.pact.test.ts`, driving a real service method.
2. Use `expectedStatus(key, CONTRACT_TARGET)` for any status that differs between targets.
3. If it needs data, use an existing state from `tests/pact/support/provider-states.ts` — or add one, and give it a handler in `scripts/verify-providers.ts` **that returns every value the interaction interpolates**.
4. Run `npm run test:pact`, then verify against the containers before publishing.

`pacts/` is generated and gitignored; `npm run test:pact` clears it first, so a renamed interaction cannot linger as a stale contract.
